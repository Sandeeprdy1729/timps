# TIMPS Governance

This document outlines the governance model for the TIMPS project.

## Overview

TIMPS uses a BDFL (Benevolent Dictator For Life) model with a team of maintainers for day-to-day operations. This approach allows for quick decision-making while ensuring community input is valued.

## Roles

### BDFL (Benevolent Dictator For Life)

The BDFL has final authority on:

- Architectural decisions
- Release cadence and versioning
- Core team appointments
- Strategic direction

Current BDFL: Sandeep Thummala (@Sandeeprdy1729)

### Maintainers

Maintainers are responsible for:

- Reviewing and merging PRs in their areas
- Responding to issues
- Ensuring code quality
- Mentoring contributors

Current maintainers are listed in [MAINTAINERS.md](./MAINTAINERS.md).

### Contributors

Anyone who contributes code, documentation, or issues is a contributor. Contributors can:

- Submit PRs
- Report bugs
- Participate in discussions

## Decision Making

### RFC Process

Major changes require an RFC (Request for Comments):

1. Create an RFC in `docs/rfcs/`
2. Community讨论 (minimum 7 days)
3. BDFL decision

### Fast Tracks

Small changes and bug fixes can be fast-tracked:

1. PR submitted
2. One maintainer approval
3. CI passes
4. Merged

## Conflict Resolution

For conflicts:

1. Discussion in issue/PR
2. Maintainer decision
3. If unhappy, escalate to BDFL

## Changing Governance

To modify this document:

1. RFC with rationale
2. BDFL approval
3. Merge

## Code of Conduct

We follow the [Contributor Covenant](https://www.contributor-covenant.org/). All participants must abide by its terms.

## Communication

- GitHub Discussions: Q&A and ideas
- GitHub Issues: Bugs and feature requests
- Discord: Real-time chat
- Twitter/X: Announcements

## Recognition

Contributors are recognized in:

- README Contributors section
- Release notes
- Monthly community calls
